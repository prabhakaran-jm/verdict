1. Executive summary

On September 19, 2020, an attacker compromised a Windows workstation (DESKTOP-SDN1RPT) and used it as a launchpad to break into the organization's domain controller (CITADEL-DC01), gaining the highest level of administrative access across the entire network. The attack began in the early hours of the morning when a hidden or already-vanished process silently launched PowerShell on the workstation, giving the attacker an interactive foothold. Shortly after, the domain controller's print service crashed under suspicious circumstances, consistent with an exploitation attempt used to help gain entry to that server. Within minutes, the attacker had connected to the domain controller over the network and created a shared folder that exposed the entire contents of the server's hard drive. The attacker then logged into the domain controller using stolen domain Administrator credentials, arriving just two minutes after that share was created. At the time the workstation's memory was captured, an active network connection to the domain controller was still open, confirming the intrusion was ongoing. The full scope of data accessed or stolen from the domain controller's exposed drive cannot be ruled out and must be assumed significant given the Administrator-level access obtained.

2. Attack narrative

At approximately 03:29 UTC on September 19, 2020, the print spooler service on the domain controller crashed with an access violation error, suggesting the attacker exploited or probed this service as a means to gain an initial foothold on the server.[F-006] At 03:34 UTC, a directory on the domain controller was configured as a network share pointing back to the entire root of the server's drive, giving anyone with access to the share unrestricted visibility into all files on the domain controller.[F-003] Just two minutes later, at 03:36 UTC, the attacker used stolen domain Administrator credentials to authenticate from the workstation (10.42.85.115) to the domain controller over the network, confirming they had already acquired the organization's most privileged account.[F-005] At 05:08 UTC, a PowerShell process with no traceable parent — indicating a hidden or already-terminated initial execution vector — spawned an interactive PowerShell session on the workstation, demonstrating the attacker maintained ongoing command execution capability well after the domain controller had been compromised.[F-002] At the time the workstation's memory was captured, an active network connection over SMB remained open between the workstation and the domain controller, consistent with the attacker continuing to browse or stage data from the fully exposed server drive.[F-003]
